## 一、项目定位

**这是什么项目**：一个支持跨宇宙（不同作品，包括同人、原创世界）的 TRPG（Tabletop Role-Playing Game）跑团知识管理系统。跑团过程中所有状态通过数据库驱动，跑团结束后结果可自动生成小说故事。

**数据库的角色**：为跑团提供单一真相源。跑团中所有查询必须通过数据库完成，不依赖人工记忆或外部文档。

**跑团模式**：**Solo Oracle-driven**（单人跑团，Oracle 规则驱动），同时保留 GM-led（有主持人）扩展空间。

**小说生成**：**半自动**——跑团会话结束后自动触发叙事管道，生成初稿草稿，人工可编辑润色。

**核心约束**：
- **认知过滤**：所有角色查询必须遵守 rp-reality-baseline 规则——角色不能知道当前时间之后的事件、不能知道其他世界的事件、不能知道超出其正典背景的信息
- **正典世界只读性**：正典世界禁止创建或修改正典事件；仅允许在正典时间线间隙中创建"跑团事件"（fanon events）
- **同人世界有限写入**：基于正典的二创世界可以修改正典，但所有偏离必须明确标记并追踪因果链影响
- **原创世界全开放**：原创世界无任何正典约束
- **统一时间轴**：所有世界通过"联合历"（Unified Calendar, UC）互相关联，每个世界定义锚点事件进行推算

**涉及的世界**（示例，展示三种类型的典型用法）：

| 世界 | 类型 | 说明 | 可创建新内容？ |
|------|:----:|------|:--:|
| `harry-potter-canon` | 正典 | 哈利波特原著世界，正典事件完全锁定 | ❌ 正典只读；仅可创建时间线间隙中的跑团事件 |
| `hp-sirius-lives` | 同人 | 衍生世界"小天狼星存活线"——从第五部神秘事务司偏离正典 | ⚠️ 允许偏离，每次偏离自动级联计算人物/政治/剧情影响 |
| `original-fantasy-world` | 原创 | 完全自创的高魔奇幻世界，无任何正典约束 | ✅ 全开放 |

## 三、动态图域

> 每个图域定义以下内容：描述 → 数据范围 → 查询能力 → 查询维度 → 关系特征 → 特殊约束。

---

### 3.1 宇宙/世界域

管理所有世界的元数据、世界之间的通道连接、以及跨世界兼容性规则。支撑跨宇宙跑团的核心域。

**描述**：世界注册信息、世界类型（正典/同人/原创）、时间线锚点、通道连接、兼容性规则。

**数据范围**：所有已注册的世界；世界间通道（包括通道稳定性、能量消耗）；跨世界物品/能力的默认兼容规则。

**查询能力**：

| # | 查询类型 | 示例问题 |
|:--:|------|------|
| W1 | 按 ID 查世界信息 | "哈利波特世界的类型和约束是什么？" |
| W2 | 查两个世界之间的通道 | "从哈利波特世界到原创奇幻世界有通道吗？代价多大？" |
| W3 | 查某世界的所有对外通道 | "从原创奇幻世界可以去哪些世界？" |
| W4 | 查某物品/能力在目标世界的兼容性 | "魔杖在新世界还能用吗？" |
| W5 | 查世界的认知映射表 | "PDA 在这个中世纪世界被理解为什么？" |
| W6 | 查统一时间轴偏移量 | "哈利波特世界的1991年对应联合历UC多少年？" |
| W7 | 按语义搜索世界 | "寻找类似武侠风格的世界" |
| W8 | 查世界层级/包含关系 | "这个多元宇宙包含哪些子世界？" |

**查询维度**：

| 维度 | 适用 |
|:--:|:----:|
| 精确查找 | ✅ |
| 关系遍历 | ✅ |
| 语义搜索 | ✅ |
| 全文搜索 | ✅ |
| 聚合/排序 | ✅ |

**约束**：
- 正典世界（canon）的元数据不可修改（名称、类型、时间线锚点）
- 通道边具有方向性：`World_A → World_B` 的稳定性可能不同于反向
- 认知映射规则按 `(源世界, 目标世界, 物品类别)` 三元组索引

**通道类型分类**：

| 通道类型 | 穿越方式 | 速度 | 举例 |
|:--------:|----------|:----:|------|
| **维度桥接** | 同一世界观内的平行维度/位面，可直接穿越或主动打开 | 实时 | 哈利波特的魔法世界↔麻瓜世界；同一个奇幻世界的不同位面 |
| **平行世界通道** | 同源/相关的平行世界之间有天然连接，也可主动发现/创造 | 实时 | 正典 `harry-potter-canon` → 同人 `hp-sirius-lives`（从某个偏离点分叉的两个世界） |
| **跨作品航行** | 完全不相关的两个宇宙（如哈利波特→星球大战），需要有载具（飞船/方舟/特殊神器），中间有航行路程 | 有航行时间 | `harry-potter-canon` → `star-wars-canon` 需要跨宇宙飞船飞行数天/周 |

### 3.2 角色域

角色身份、物理特征、社会关系、能力标签和认知范围。这是跑团中查询最密集的域。

**描述**：角色的完整档案，分四个层级——

**① 身份层（存于 Character 顶点）**：
`name`, `aliases`, `character_origin`（**canon** / **fanon** / **original**）,
`is_party_member`, `party_join_date`,
`species`（人类/精灵/突变体等）, `birth_date`（本地历）,
`gender`, `sexuality`, `chronological_age`, `apparent_age`,
`height`, `hair_color`, `distinguishing_features`, `appearance_summary`, `typical_attire`,
`nationality`, `social_class`, `occupation`,
`origin_world`（原生世界）

> Character 顶点保存**当前外貌**，直接读取即可。`CharacterSnapshot` 同时保存每次变化的历史快照——查当前走 Character，查历史走 Snapshot。

**② 能力层（纯标签系统）**：
采用纯标签系统。角色的能力用描述性短句定义，无数值。示例：`"精通黑魔法防御术"`、`"力大无穷"`、`"擅长潜行"`。
标签在跑团中通过 Oracle 判定来检定——有关联标签则有利修正，无关联标签则中性，有冲突标签则不利修正。

**③ 跨世界层**：
`languages_spoken`（会说什么语言）, `world_travel_history`（去过哪些世界）,
`default_cognition`（来自原生世界的认知基线——如：来自赛博朋克世界→对魔法一无所知，看见悬浮咒以为是黑科技）

**④ 叙事层**：
`personality_tags`（性格标签 [`冲动`, `忠诚`, `多疑`]——Oracle 推演行为时参考）,
`core_motivation`（核心动机——角色想要什么：回家/复仇/寻找某人）,
`secret`（秘密背景——可能被 NPC 触发的剧情伏笔）,
`voice`（说话方式——如"说话简短粗暴，用词粗俗但偶尔冒出古老的精灵谚语"）,
`speech_constraints`（禁忌话题列表——如`["不在精灵面前说脏话", "永远不会提导师的死"]`）,
`background`（完整背景故事——长文本，Oracle 推演角色行为时的参考）,
`voice_profile`（声线资料 JSON——未来 TTS 用，含 `timbre`, `pitch_range`, `pace`, `accent`, `emotional_default`, `sample_ref`）

**角色分类矩阵**：

| | 正典 (canon) | 原创 (original) | 同人版本 (fanon) |
|------|:--:|:--:|:--:|
| **主角团** | 仅在同人世界中允许 | ✅ 最常见 | ✅ |
| **非主角团** | 标准 NPC 角色 | 原创配角 | 同人额外角色 |
| **跨世界** | 允许，需认知映射判定 | ✅ 全开放 | ✅ 全开放 |
| **属性改** | 正典属性锁死，可追加跑团经历 | 完全自由 | 受限自由 |

> **核心规则**：跨世界传送只迁移 `is_party_member=true` 的角色。非主角团位置不变。NPC可在跑团后期通过剧情触发加入主角团，`party_join_date` 记录加入时间。

**⑤ 认知层（三维认知模型）**：

角色认知 = **世界层** + **角色层** + **经历层** 三层叠加。

**世界层——社会认知模板**：每个世界的不同社会阶层有默认认知边界。

| 阶层 | 地理认知范围 | 知识范围 | 谣言接收率 |
|:----:|------|------|:---------:|
| 农民 | 本村+邻镇 | 基本农耕、本地宗教 | 高 |
| 市民 | 本城+周边 | 行业知识、城中新闻 | 中 |
| 贵族 | 全国+邻国 | 政治、历史、纹章学、外语 | 中 |
| 学者 | 不限 | 文献记载范围内的知识 | 低 |

角色创建时按 `social_class` + `world_of_origin` 从世界认知模板中继承基础认知。

**角色层——个人认知**：正典背景/原创经历赋予的特定知识。如：赫敏知道密室事件始末。

**经历层——动态增长**：跑团互动中亲眼所见/亲耳所闻的事实。记录每次认知获取：

| 字段 | 说明 |
|------|------|
| `fact_id` | 认知事实 ID |
| `fact_type` | event / item / character / location / concept |
| `source_type` | **亲眼所见** / 道听途说 / 文献记载 / 推测 |
| `confidence` | 可信度 0-100（亲眼最高，谣言最低） |
| `acquired_session` | 在哪个跑团会话中获得 |

**认知可信度**：信源质量影响角色是否能准确判断。谣言可能导致错误行动，产生戏剧冲突。

**认知冲突检测**：当角色先知道错误版本、后知道真相时，系统标记冲突并记录角色当前相信哪个版本。错误信念保留用于叙事回溯（"我曾经以为……"）。

**跨世界认知断层**：角色穿越到新世界后，面临认知框架完全不匹配。系统记录 `unfamiliar_concepts`（无法理解的概念列表），角色需通过多个事件逐步适应（如黑客第一次见魔法→尝试用纳米粒子解释→逐渐接受魔法存在）。被证伪的旧认知触发认知冲突检测。

**小说生成中的认知过滤**：每段叙事只呈现当前 POV 角色认知范围内的事物，不越界。

**⑥ 腐蚀层（可选，40K 混沌腐蚀专用）**：

仅在使用混沌/克苏鲁/黑暗奇幻等存在腐蚀机制的世界时启用。普通世界角色此项为空。

| 字段 | 说明 |
|------|------|
| `corruption_level` | 腐蚀值 0-100。0=纯洁，30=低语开始，60=明显变异，100=完全堕落 |
| `mutation_manifestations` | 已显现的变异列表，如 `["第三只眼", "皮肤硬化"]` |
| `heresy_conviction` | 异端信念强度 0-100。腐蚀影响认知——被腐蚀的角色会合理化极端行为 |

腐蚀值通过 Oracle 触发事件积累：接触混沌遗物 +5、目睹恶魔 +10、被灵能反噬 +d20。每达到阈值（30 / 60 / 90）自动触发变异表检定。

**数据范围**：所有参与跑团的 PC 和 NPC；他们之间的关系（KNOWS 边）；角色在每个时间点的状态快照。

**查询能力**：

| # | 查询类型 | 示例问题 |
|:--:|------|------|
| C1 | 按 ID 查完整角色画像 | "哈利·波特的样貌、种族、年龄、衣着、标签能力、当前装备是什么？" |
| C2 | 查两个角色之间的关系 | "哈利对斯内普的信任度/好感度是多少？" |
| C3 | 查某个角色的关系网（过滤） | "哈利最信任的三个朋友是谁？" |
| C4 | 按语义搜索角色 | "寻找擅长黑魔法防御术的角色" |
| C5 | 查任意历史时间点的角色状态 | "第三学年开始时哈利的状态快照是什么？" |
| C6 | 查角色状态变化日志 | "哈利的哪个标签是在哪个事件后获得的？" |
| C7 | 查角色归属 | "哈利属于哪个学院/派系？" |
| C8 | 查角色当前位置 + 轨迹 | "哈利当前在哪里？之前去过哪些地方？" |
| C9 | NPC对角色第一印象判定 | "一个猎魔人世界的人看到一个东亚面孔会怎么反应？" |
| C10 | 按外貌特征搜索 | "寻找红色头发、脸上有疤的角色" |

**查询维度**：

| 维度 | 适用 |
|:--:|:----:|
| 精确查找 | ✅ |
| 关系遍历 | ✅ |
| 语义搜索 | ✅ |
| 全文搜索 | ✅ |
| 时间区间查询 | ✅ |
| 聚合/排序 | ✅ |

**关系特征**：
- 关系是**不对称的**：A→B 的 trust/attraction/respect 可能不同于 B→A
- 关系变化**保留历史**：通过 RelationshipEvent 记录每次变化
- 关系有**时间区间**：使用 unified_since / unified_until 标记有效区间
- 关系有**阶段语义**：KNOWS 边补充 `relationship_phase` 字段——`"相识"` / `"命运绑定"` / `"若即若离"` / `"已破裂"` / `"至死不渝"`。阶段变化触发 RelationshipEvent 记录。

**约束**：
- 所有角色查询必须携带**认知过滤**——不能返回角色当前不知道的信息
- 同一角色在不同世界有不同身份映射（见跨世界桥接域）

### 3.3 派系域

管理组织、势力的层级结构和外交关系。

**描述**：派系/组织的元数据、层级树、盟约/敌对关系、控制的地盘范围。

**数据范围**：各大组织/势力（如霍格沃茨、凤凰社、食死徒）；派系内部层级；派系间外交状态。

**查询能力**：

| # | 查询类型 | 示例问题 |
|:--:|------|------|
| F1 | 按 ID 查派系信息 | "凤凰社的宗旨和成员构成是什么？" |
| F2 | 查两个派系的关系 | "凤凰社和魔法部当前的关系友好还是敌对？" |
| F3 | 查派系成员（过滤+排序） | "食死徒中战斗力最高的五个成员是谁？" |
| F4 | 查派系控制区域 | "食死徒控制了多少地区？" |
| F5 | 查派系层级 | "霍格沃茨的指挥链是怎样的？" |
| F6 | 查外交变化历史 | "凤凰社和魔法部的关系是在哪次事件后恶化的？" |
| F7 | 查派系资源 | "每个派系掌握的关键资源/神器有哪些？" |
| F8 | 查历史势力版图 | "战争爆发前食死徒控制了多少地盘？" |

**查询维度**：

| 维度 | 适用 |
|:--:|:----:|
| 精确查找 | ✅ |
| 关系遍历 | ✅ |
| 语义搜索 | ✅ |
| 时间区间查询 | ✅ |
| 聚合/排序 | ✅ |

**关系特征**：
- 派系间关系是**不对称的**（A 可能视 B 为盟友，B 可能视 A 为威胁）
- 成员归属关系有**时间区间**（入团/退团时间）
- 派系支持**父子层级嵌套**（如：凤凰社 ⊂ 邓布利多军 ⊂ 霍格沃茨），通过 `hierarchy_parent` 实现树形结构
- **影响力模型**：数值制（0-100），一个派系在每个区域有独立影响力值，多个派系可同时影响同一区域

**战役/冲突追踪**：动态军事冲突建模。
- `Conflict` Vertex：`type`(战争/围城/游击), `belligerents`派系列表, `front_line`战线位置, `status`(准备中/进行中/停火/已结束), `controlled_zones`控制区域变化列表
- 关键边：`CONTENDED_BY` (Conflict→Faction), `BATTLEFIELD` (Conflict→Location)

**外交关系细化**（复杂政治场景可选）：
`NEGOTIATES_WITH` 边替代 ALLIED_WITH/HOSTILE_WITH 二元关系。字段：`diplomatic_status`(war/ceasefire/non_aggression/defensive_pact/temporary_truce/vassal/tributary), `status_since`, `mutual_threat`(共同敌人), `ideological_alignment`(-100~+100)

### 3.4 地点空间域

空间拓扑、路线导航、交通工具管理和位置追踪。

#### 三层判定机制（详程）

##### 第一层：通道存在性检查

物品/角色要穿越到新世界，首先验证源世界→目标世界之间是否有可行的通道。

**判定输入**：
```json
{
  "source_world": "harry-potter-canon",
  "target_world": "original-fantasy-world",
  "entity_type": "item",
  "entity_id": "wand_01"
}
```

**判定逻辑**：
```
1. 查询 CONNECTS_TO 边：World(源)→World(目标)
2. 若无通道 → 返回 DISABLED（无法穿越）
3. 若有通道：
   查 stability 字段（1-10）：
   ≤ 3 → 极不稳定，穿越失败率 60%
   4-6 → 中等稳定，消耗额外能量
   ≥ 7 → 稳定通道，正常穿越
4. 查 one_way：true=仅单向 / false=双向通行
```

**判定输出**：
```json
{
  "channel_exists": true,
  "stability": 8,
  "energy_cost": "low",
  "one_way": false,
  "pass_channel": true
}
```

##### 第二层：兼容性等级判定

通过通道后，物品/能力在新世界中的运作状态判定。

**四种兼容性等级**：

| 等级 | 含义 | 示例 |
|:----:|------|------|
| **full** | 完全正常运作，功能不变 | 普通冷兵器在任何世界都能正常使用 |
| **degraded** | 降级运作，效能减弱但原理不变 | 魔杖在无魔世界中只能当普通短棍 |
| **reinterpreted** | 被新世界规则重新解释，表现改变 | PDA 在魔法世界被理解为"预言水晶" |
| **disabled** | 完全禁用，无法运作 | 魔法物品在纯科学宇宙中失效 |

**判定逻辑**（以魔杖穿越为例）：
```
1. 查 COMPATIBILITY_RULE 边：(源世界, 目标世界, 物品类别)
   └─ 有预定义规则 → 直接返回

2. 无预定义规则 → 自动推断：
   比较 power_source:
   ├─ 相同 → full
   ├─ 兼容（magic ↔ mana）→ degraded（需转化）
   └─ 不兼容（magic ↔ science）→ disabled
   比较 technology_level:
   ├─ 目标 > 源 → reinterpreted（被理解为更高级事物）
   └─ 目标 < 源 → degraded（被理解为更原始事物）

3. 特殊豁免（如"该物品是神器"）→ 提升一级
```

**判定输出**：
```json
{
  "compatibility": "degraded",
  "effect": "魔杖在 mana 世界中需消耗持有者法力值来驱动",
  "stat_modifiers": { "power": -2, "duration": "halved" }
}
```

##### 第三层：认知映射

目标世界的原生角色在看到外来物品时默认如何理解。

**映射数据结构**：
```json
{
  "entry_id": "bridge_wand_01",
  "source_world": "harry-potter-canon",
  "target_world": "original-fantasy-world",
  "item_name": "holly_wand",
  "local_name": "魔杖",
  "local_understanding": "一种通过 mana 施法的法杖，长约11英寸",
  "misconceptions": ["当地人认为只有巫师才能使用"],
  "reveal_triggers": ["角色在当地人面前展示魔杖时触发默认映射"]
}
```

**映射规则**：
- **默认映射**：目标世界角色自动按 `cognitive_mapping` 理解，无须检定
- **突破映射**：若外来角色主动解释真相，目标角色需过感知检定才能接受
- **三层理解**：一级=表面 \| 二级=功能 \| 三级=本质

#### 运行时判定总流程

```
角色/物品触发穿越
  │
  ├─ 阶段1：通道检查
  │   查 CONNECTS_TO(源→目标)
  │   ├─ 无通道 → ❌ disabled
  │   └─ 有通道 → 稳定性能量评估
  │
  ├─ 阶段2：兼容性判定
  │   查 COMPATIBILITY_RULE(源,目标,类别)
  │   └─ 自动推断或预定义规则
  │   └─ 结果: full/degraded/reinterpreted/disabled
  │
  ├─ 阶段3：认知映射
  │   查 COGNITIVE_MAP(源,目标,物品)
  │   └─ 获取 local_name + understanding + misconceptions
  │
  └─ 阶段4：状态更新
      ├─ 更新 current_world 为目标世界
      ├─ 设置 active_compatibility 等级
      ├─ 应用 stat_modifiers
      └─ 存入穿越记录（CROSSED_THROUGH 边）
```

- `abilities`（特殊能力：隐形/毒素/飞行）
- `typical_bounty`（标准赏金）
- `loot`（战利品：突变诱发物/皮毛/牙齿）

**关键边**：
- `HABITAT_IN` (Monster→Location): 栖息地范围
- `HUNTED_BY` (Monster→Character): 被谁狩猎过
- `DROPS` (Monster→Item): 掉落物

### 3.10 物品/装备域

物品实例化——支持持有追踪、消耗品状态、跨世界实例规则、历史传承。

**描述**：所有可持有、可使用的物品实例，含装备、消耗品、神器、任务道具。

**查询能力**：

| # | 查询类型 | 示例问题 |
|:--:|------|------|
| I1 | 查物品详情 | "这把剑的属性和当前状态是什么？" |
| I2 | 查物品当前持有者 | "霜之哀伤现在在谁手里？" |
| I3 | 查物品历史 | "这把剑之前被谁持有过？经历了哪些事件？" |
| I4 | 查物品跨世界兼容性 | "这把魔法剑在无魔世界还能用吗？" |
| I5 | 按类别/属性过滤 | "所有远程武器有哪些？" |
| I6 | 查消耗品余量 | "这发能量电池还剩多少？" |

**关键属性 (Item Vertex)**：
- `item_id`, `name`, `category`(weapon/armor/consumable/artifact/misc)
- `properties`(JSON: damage/ammo/element/weight 等)
- `origin_world`, `description`
- `quantity`(可堆叠物品的数量)
- `condition`(正常/损坏/需修理)

**关键边**：
- `EQUIPPED_BY` (Item→Character): 当前持有者
- `DROPS` (Monster→Item): 怪物掉落
- `CREATED_AT` (Item→Event): 制造/获得事件
- `BELONGS_TO` (Item→Faction): 所属组织
- `TRANSFERRED_TO` (Character→Character, on Item): 物品转让，含 `item_ref`, `unified_time`, `trigger_event`——可回溯"这把刀经过谁的手"

## 四、文本与知识域

长篇文本支撑设定查询和语义回溯。每个文本域支持**混合检索**——单次查询同时走语义向量和关键词精确匹配，服务端融合排序。

| 域 | 内容说明 | 检索需求 | 向量化 |
|------|------|------|:-----:|
| 世界观设定 | 各世界的设定文档、历史、文化、地理 | 语义搜索 + 关键词搜索 | ✅ |
| 跑团日志 | 每次跑团会话的叙事正文 + 对话转录 | 语义搜索 + 关键词搜索（仅叙事正文） | ✅ |
| 小说章节 | 自动生成或人工编写的小说章节省略 | 语义搜索 + 关键词搜索 | ✅ |
| 规则文档 | Oracle 规则说明、检定系统文档 | 全文搜索为主，语义搜索为辅 | ❌ |

**检索策略**：
- **混合检索**：向量 + BM25 全文，使用 RRF（Reciprocal Rank Fusion）服务端融合
- **向量化范围**：世界观设定、跑团日志（仅叙事正文）、小说章节需要向量化
- 对话转录（transcript）不参与向量搜索
- 规则文档以全文搜索为主，不需要向量化

### 小说生成管道

**触发时机**：每次跑团会话结束后自动触发生成。不依赖于工触发或批量积累。

跑团日志 → 小说章节的转换流程：

**风格模板机制**：跑团开始前（或在运行状态中随时切换），选一种叙事风格模板，生成时按模板渲染。

| 风格模板 | 适用场景 | 输出示例 |
|:--------:|----------|----------|
| **第一人称日记** | 侧重内心独白、角色心理 | "我从未想过会走进那扇门……" |
| **第三人称史诗** | 侧重外部动作、战斗、大场景 | "哈利·波特举起魔杖，一道绿光划过天际。" |
| **轻小说风格** | 轻松叙事、对话密集、有旁白吐槽 | "如果要用一句话说明今天的遭遇——那就是糟透了。" |

生成流程：

```
跑团日志（原始）
  → 步骤1：叙事过滤
      提取关键场景（剧情线权重高的事件）
      过滤掉无聊的检定过程和重复对话
  → 步骤2：视角选择
      选择 POV 角色（遵守角色认知过滤）
      按当前风格模板选择视角数量
      （第一人称=固定单POV，第三人称/轻小说=可多POV）
  → 步骤3：风格渲染
      应用选定的风格模板
      保持角色声线一致（viola voice_check）
  → 步骤4：正典校验
      检查生成内容不违反任何正典设定
      标记任何潜在的偏离以供人工审核
  → 步骤5：章节输出
      保存为小说章节文档
      状态标记为 draft（草稿）
```

**小说章节文本域额外字段**：
- `pov_character` — 本章视角角色
- `style` — 叙事风格（diary / epic / light_novel）
- `corresponding_session` — 对应的跑团会话 ID
- `canon_checked` — 正典校验是否通过
- `status` — draft（草稿）/ published（已完成）

## 六、关键机制

> 以下机制是跑团系统的核心功能组件。

### 6.1 跨世界能力兼容

物品/能力进入新世界时的**三层判定机制**（定义及详程见 [3.7 跨世界桥接域](#37-跨世界桥接域)）：

1. **通道存在性**：源世界→目标世界是否有稳定通道？能量消耗多少？
2. **兼容性等级**：full / degraded / reinterpreted / disabled
3. **认知映射**：目标世界的原生角色默认如何理解这个外来的物品/能力

> 完整的输入输出数据格式和自动推断逻辑见 [三层判定详程](#三层判定机制详程)。

### 6.2 认知过滤

每个角色认知 = **世界层**（社会阶层认知边界）+ **角色层**（个人背景）+ **经历层**（跑团见闻）三层叠加。

**基础校验规则**：
- 不能知道当前时间之后的事件（时间隔离）
- 不能知道其他世界的事件（世界隔离）
- 不能知道超出其正典背景的信息（背景隔离）
- 正典中 pending（未发生）的事件不能提及

**增加的三层校验**：

| 层级 | 规则 | 举例 |
|:----:|------|------|
| **世界层** | 按社会阶层继承该世界认知模板上限 | 农民不会知道宫廷密谋 |
| **可信度** | 按来源传递置信度；谣言不触发剧情线推进 | 听说有战舰≠确认存在 |
| **认知冲突** | 高可信证据推翻低可信认知时标记冲突 | 谣言"是恶魔"→亲眼见救人→修正信念 |
| **跨世界断层** | 原世界认知框架不适用新世界概念，记录为 `unfamiliar_concepts` | 黑客第一次见魔法→尝试用纳米粒子解释 |

**实现方式**：所有角色查询携带认知过滤器。认知记录存入 `CognitionRecord` Vertex。

### 6.3 统一时间轴

多世界项目需要将各自历法统一到一条时间线。

**方法**：联合历（Unified Calendar, UC）
- 每个世界定义一个或多个锚点事件（如：哈利波特世界的 "霍格沃茨之战" = UC+9960）
- 本地时间 ↔ UC 通过锚点偏移量推算
- Edge 边上的时间戳使用 unified_since / unified_until 双字段

### 6.4 正典偏离追踪与级联计算

当同人世界（fanon）在某一节点偏离正典时，系统需从**人物、政治、剧情**三个维度进行级联影响计算。

> **偏离粒度**：支持**事件侧面级偏离**——可只偏离事件的某个侧面（如参与者不同但结果不变，或时间偏移但内容不变），而非要求整个事件全改或不改。偏离的具体变动范围记录在 `DeviationRecord.changed_aspects` 字段中。

#### 6.4.1 偏离的三级影响传播

| 影响维度 | 需要重新计算什么 | 数据库遍历路径 |
|------|------|------|
| **人物影响** | 谁会死/谁会活 → 关系网重算（信任、敌对、羁绊） → 角色属性变化 → 角色状态快照更新 | `CanonEvent →[PARTICIPATED_IN]→ Character →[KNOWS]→ Character[...]` |
| **政治影响** | 势力版图洗牌 → 遗产归属变化 → 派系实力重估 → 外交关系调整 | `Character →[BELONGS_TO]→ Faction →[CONTROLS]→ Location` + `Faction →[ALLIED_WITH/HOSTILE_WITH]→ Faction` |
| **剧情影响** | 该事件消失/改变 → 依赖它的下游事件全部标记待处理 → 因果链断裂需重新桥接 → 剧情线走向推导 | `Event →[CAUSES]→ Event[...] →[ADVANCED_BY]→ PlotThread` |

#### 6.4.2 偏离的级联计算流程

```
事件偏离触发（同人世界）
  │
  ├─ 阶段0：偏离合法性检查
  │   ├─ 正典世界 → ❌ 拒绝偏离（事件只读）
  │   └─ 同人世界 → ✅ 允许偏离，进入级联计算
  │
  ├─ 阶段1：标记偏离点
  │   ├─ 更新 CanonEventStatus.status = modified/skipped
  │   ├─ 记录 divergence_reason、occurred_in_chapter
  │   └─ 创建 DeviationRecord 顶点记录本次偏离
  │
  ├─ 阶段2：识别受影响域
  │   ├─ 人物 → 沿 PARTICIPATED_IN 边找出参与该事件的角色
  │   │           再沿 KNOWS 边展开受影响的关系网
  │   ├─ 派系 → 沿角色的 BELONGS_TO 上溯到派系
  │   │           再沿 CONTROLS/ALLIED_WITH 展开势力影响
  │   └─ 剧情 → 沿 CAUSES 边向下游 BFS 遍历所有依赖事件
  │           标记所有受影响的事件为 affected
  │
  ├─ 阶段3：逐域重算
  │   ├─ 人物状态 + 关系网重新推导
  │   ├─ 势力版图重新计算
  │   └─ 因果链替换 → 断裂处标记为 broken
  │       (断裂的因果链需要跑团者/Oracle 决定新的走向)
  │
  └─ 阶段4：生成偏离报告
      ├─ 记录"这个世界从哪开始不同了"
      ├─ 输出对小说生成有用的上下文说明
      └─ 偏离报告存入 DeviationRecord（偏离记录）
```

#### 6.4.3 查询能力

| # | 查询类型 | 示例问题 |
|:--:|------|------|
| D1 | 查单个偏离点的详情报 | "这个事件被改成了什么？为什么改？" |
| D2 | 查偏离的人物影响范围 | "这个偏离导致哪些角色的状态发生了变化？" |
| D3 | 查偏离的政治影响范围 | "这个偏离导致哪些派系的势力版图发生了变化？" |
| D4 | 查偏离的剧情影响范围 | "这个偏离导致哪些下游事件被标记为 affected？" |
| D5 | 查同人世界的偏离历史 | "这个同人世界目前有多少个偏离点？累计影响范围多大？" |
| D6 | 查偏离的因果链断裂点 | "这个偏离导致哪些因果链断开了？需要重新桥接？" |
| D7 | 查所有未处理的受影响事件 | "哪些下游事件因为偏离被标记为 affected 但还未重新推导？" |
| D8 | 查偏离报告 | "这个同人世界与正典的差异摘要是什么？" |

#### 6.4.4 正典世界 vs 同人世界的偏离策略对比

| 维度 | 正典世界 (canon) | 同人世界 (fanon) |
|------|:--:|:--:|
| 能否创建新事件 | ✅ 仅能在时间线间隙创建 | ✅ 自由创建 |
| 能否修改正典事件 | ❌ 绝对锁定 | ✅ 可以改写 |
| 正典事件的状态 | 永远 `triggered` | 允许 `modified/skipped` |
| 偏离级联计算 | 不适用（无偏离） | 触发 4 阶段级联 |
| 因果链 DAG | 始终完整，由正典定义 | 可能断裂，需重新桥接 |
| 偏离报告 | 始终为空 | 每次偏离生成 1 份报告 |

### 6.5 历史状态查询

任意时间点的完整状态快照——关系、物品、位置、势力版图。

**实现方式**：所有 Edge 和状态属性使用 unified_since / unified_until 时间区间化。
查询方式：
- `WHERE unified_since <= X AND (unified_until IS NULL OR X < unified_until)`

### 6.6 小说生成机制

跑团会话结束后自动触发叙事管道，将原始日志转化为小说故事。

**触发条件**：
- 一次跑团会话结束（手动结束或关键节点到达）
- GM/系统判定当前剧情达到可写的叙事节点

**叙事管道**（详见 [四、文本与知识域](#四文本与知识域) 的小说生成管道）：

| 阶段 | 输入 | 输出 | 关键逻辑 |
|:----:|------|------|------|
| 叙事过滤 | 原始跑团日志 | 关键场景列表 | 筛选剧情线权重高的事件、角色关键对话 |
| 视角选择 | 关键场景 | 场景 + POV 分配 | 遵守认知过滤，每个场景 1-2 个 POV |
| 风格渲染 | 场景 + POV | 叙事草稿段落 | 保持角色声线，应用选定风格 |
| 正典校验 | 草稿段落 | 校验报告 | 检查是否违反正典设定 |
| 章节编排 | 校验后的段落 | 小说章节文档 | 时间排序，分章，添加元数据 |

**输出标记**：
- 生成的章节标记为 `draft` 状态
- 包含 `pov_character`、`style`、`corresponding_session` 元数据
- 通过正典校验的标记 `canon_checked: true`
- 人工编辑后改为 `published` 状态

## 八、数据持久化与双写

### 原则

- **数据库是唯一权威源**。运行时所有读写走 ArcadeDB，.md 文件是持久化快照
- **写顺序**：先写数据库，再更新本地 .md 文件
- **冲突裁决**：.md 与 DB 不一致时，以 DB 为准

### DB → .md（导出）

每次跑团会话结束后，自动将所有变更导出为结构化 .md 文件。组织方式：

```
data/
  characters/{world}/     -- 按世界分目录
  factions/{world}/
  locations/{world}/
  events/{world}.md       -- 时间线单文件
  items/{world}/
  monsters/{world}/
  contracts/{world}/
  sessions/{session_id}.md
```

每条记录用 YAML 前置元数据 + Markdown 正文格式：

```yaml
---
id: "char_geralt"
world: "witcher"
type: "character"
origin: "canon"
updated: "UC+11945.3"
---
**基本资料**
种族：猎魔人（突变体）
外貌：白发、猫眼、两道剑
**标签**
["猎魔人感知", "剑术大师", "炼金术"]
```

### .md → DB（导入/恢复）

启动时扫描 `data/` 目录，解析 YAML+MD → 写入 ArcadeDB。

| 模式 | 行为 |
|:---:|------|
| **增量（默认）** | 已有记录按 id 去重跳过，不覆盖 DB 中已有数据 |
| **强制覆盖** | DB 中已有记录用 .md 内容覆盖。用于 DB 重置后完整恢复 |
| **差异合并** | 按 updated 字段比较，取最新版本 |

### 目录维护

- `data/` 目录应纳入 Git 版本控制
- 会话日志（`sessions/`）按时间排序，不可手动修改
- 设定数据（`characters/factions` 等）可直接手工编辑，下次导入时增量合并

### 按 Session 回滚

如需撤销某次跑团会话的全部改动：

| 数据类型 | 删除条件 |
|------|------|
| Event（跑团事件） | `event_id` 包含 session 标识 |
| CognitionRecord | `acquired_session = "session_X"` |
| CharacterSnapshot | `chapter_ref` 关联本次 session |
| SessionLog | `session_id = "session_X"` |
| CanonEventStatus | `occurred_in_chapter` 回滚，原地重建 |
| DeviationRecord | `occurred_in_chapter` |
| NovelChapter | `corresponding_session = "session_X"` |

**关系边回滚**（需要 `closed_by_session` 字段）：
- 删除本次创建的新边：`DELETE WHERE closed_by_session = "session_X"`
- 恢复被本次关闭的旧边：`SET unified_until = NULL WHERE closed_by_session = "session_X"`

> 所有带 `unified_until` 的边记录 `closed_by_session` 字段，否则无法分辨"被这次关闭还是更早关闭"。

### Turn 级微回滚

Agent 回复不满意时，可回滚单个 Turn 而非整个 Session。机制：每次 Agent 推理前保存 Checkpoint。

